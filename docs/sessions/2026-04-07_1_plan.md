# 현명 소비 시스템 고도화
> 날짜: 2026-04-07 | 회차: 1 | 상태: 기획

## 요청 내용
- 전주 대비 현명 소비 (특정 기간 평균값으로 비정상적 금액 판별)
- 현명 소비 Level, Character, Statistics
- 무소비 Challenge

## 현재 상태 분석

### 부족한 점 (선행 작업 필요)
1. **Transaction에 거래 타입 없음** — 백엔드에 INCOME/OUTCOME/REFUND 구분 필드 없음 (FE만 하드코딩)
2. **통계/분석 도메인 없음** — 기존 코드에 스케줄러/배치/통계 관련 코드 전무
3. **금액 필드가 Int** — 음수/양수 구분 또는 타입 필드로 소비/수입 구분 필요

### 기존 인프라 활용 가능
- TransactionRepository의 `findByMemberIdWithFilters()` — 날짜/카테고리/태그 필터
- Notification 시스템 — 알림/챌린지 결과 전달 활용 가능
- Kafka — 비동기 이벤트 처리

## 성능 설계

### 데이터 접근
- **주간 통계**: 주 1회 집계 → `WeeklyStatSnapshot` 테이블에 저장 → 조회 시 스냅샷만 읽기 (거래 테이블 직접 조회 안 함)
- **이상 소비 감지**: 최근 8주 스냅샷의 평균/표준편차 계산 (고정 건수, O(1) 조회)
- **레벨 조회**: `SpendingLevel` 테이블 PK 조회 (O(1))
- **무소비 판정**: `SELECT COUNT(*) FROM transactions WHERE member_id=? AND date=? AND tx_type='OUTCOME' AND deleted=false` — 인덱스 활용

### 예상 규모
- 회원당 주간 거래: ~20-50건
- 주간 스냅샷: 회원 수 × 52주/년 (수백 건 수준)
- 챌린지: 회원당 동시 1개 제한

### 리소스 제약
- Synology NAS: 메모리 제한 있음 → 배치 사이즈 100건 이하
- MySQL 3307: 커넥션 풀 기본값 유지
- Kafka: 기존 토픽 활용 또는 새 토픽 추가

### 설계 결정
- **스냅샷 패턴**: 실시간 집계 대신 주기적 스냅샷 → 조회 성능 보장
- **이벤트 기반 경험치**: 거래 생성 시 동기 처리 (Kafka 불필요, @TransactionalEventListener)
- **챌린지 판정**: @Scheduled 크론으로 매일 자정 배치 (소규모 데이터이므로 단일 쿼리로 충분)

## DB 스키마 설계 (초안)

### transactions 테이블 변경
```sql
ALTER TABLE transactions ADD COLUMN tx_type VARCHAR(20) NOT NULL DEFAULT 'OUTCOME';
-- 기존 데이터: 모두 OUTCOME으로 설정 (이후 수동 분류 가능)
```

### weekly_stat_snapshots (신규)
```sql
CREATE TABLE weekly_stat_snapshots (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    member_id BIGINT NOT NULL,
    week_start_date DATE NOT NULL,        -- 해당 주 월요일
    week_end_date DATE NOT NULL,          -- 해당 주 일요일
    total_outcome INT DEFAULT 0,          -- 주간 총 지출
    total_income INT DEFAULT 0,           -- 주간 총 수입
    total_refund INT DEFAULT 0,           -- 주간 총 환불
    transaction_count INT DEFAULT 0,      -- 거래 건수
    avg_daily_outcome INT DEFAULT 0,      -- 일평균 지출
    top_category_id BIGINT,               -- 최다 소비 카테고리
    is_anomaly BOOLEAN DEFAULT FALSE,     -- 이상 소비 여부
    anomaly_ratio DECIMAL(5,2),           -- 평균 대비 비율 (e.g., 1.5 = 150%)
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_member_week (member_id, week_start_date),
    INDEX idx_member_id (member_id)
);
```

### spending_levels (신규)
```sql
CREATE TABLE spending_levels (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    member_id BIGINT NOT NULL UNIQUE,
    level INT DEFAULT 1,
    experience INT DEFAULT 0,
    character_type VARCHAR(50) DEFAULT 'SEED',  -- SEED, SPROUT, TREE, FOREST 등
    total_saving_days INT DEFAULT 0,            -- 누적 무소비 일수
    current_streak INT DEFAULT 0,               -- 현재 연속 무소비 일수
    best_streak INT DEFAULT 0,                  -- 최고 연속 무소비 일수
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_member_id (member_id)
);
```

### challenges (신규)
```sql
CREATE TABLE challenges (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    member_id BIGINT NOT NULL,
    type VARCHAR(30) NOT NULL DEFAULT 'NO_SPEND',  -- NO_SPEND, BUDGET_LIMIT 등
    status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',   -- ACTIVE, COMPLETED, FAILED
    start_date DATE NOT NULL,
    end_date DATE NOT NULL,
    target_days INT NOT NULL,                        -- 목표 무소비 일수
    achieved_days INT DEFAULT 0,                     -- 달성 무소비 일수
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_member_status (member_id, status)
);
```

### challenge_daily_logs (신규)
```sql
CREATE TABLE challenge_daily_logs (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    challenge_id BIGINT NOT NULL,
    log_date DATE NOT NULL,
    is_no_spend BOOLEAN NOT NULL,
    outcome_amount INT DEFAULT 0,                    -- 당일 지출 금액 (0이면 무소비)
    UNIQUE KEY uk_challenge_date (challenge_id, log_date),
    INDEX idx_challenge_id (challenge_id)
);
```

## 레벨 시스템 설계 (초안)

### 경험치 획득 조건
| 행동 | 경험치 | 비고 |
|------|--------|------|
| 거래 등록 | +10 | 하루 최대 5회 |
| 무소비 달성 (1일) | +20 | 챌린지 진행 중일 때 |
| 전주 대비 소비 감소 | +30 | 주간 비교 시 |
| 이상 소비 없음 (1주) | +15 | 주간 정산 시 |
| 챌린지 완료 | +100 | 목표 달성 시 |

### 레벨 구간
| 레벨 | 필요 경험치 | 캐릭터 | 설명 |
|------|------------|--------|------|
| 1 | 0 | 씨앗 (SEED) | 시작 |
| 2 | 100 | 새싹 (SPROUT) | 소비 습관 형성 중 |
| 3 | 300 | 묘목 (SAPLING) | 꾸준한 관리 |
| 4 | 600 | 나무 (TREE) | 안정적 소비 패턴 |
| 5 | 1000 | 숲 (FOREST) | 현명한 소비 마스터 |

### 이상 소비 감지 기준
- **기준**: 최근 8주 평균 주간 지출의 **1.5배 초과** 시 "이상 소비"로 판정
- **표시**: 주간 비교 화면에서 경고 아이콘 + 초과 비율 표시
- **제외**: 거래 건수 3건 미만인 주는 통계에서 제외 (데이터 부족)

## API 설계 (초안)

### 주간 비교
```
GET /api/v1/statistics/weekly-comparison?weekOffset=0
Response: {
  currentWeek: { startDate, endDate, totalOutcome, totalIncome, txCount, avgDaily, topCategory },
  previousWeek: { ... },
  changeRate: -12.5,          // 전주 대비 변화율 (%)
  isImproved: true,           // 전주 대비 개선 여부
  isAnomaly: false,           // 이상 소비 여부
  anomalyRatio: 0.87          // 평균 대비 비율
}
```

### 소비 통계 요약
```
GET /api/v1/statistics/summary
Response: {
  level: { level: 3, exp: 450, nextLevelExp: 600, characterType: "SAPLING" },
  streak: { current: 5, best: 12, totalSavingDays: 45 },
  thisMonth: { totalOutcome: 850000, totalIncome: 3000000, txCount: 23 },
  categoryBreakdown: [ { category: "식비", amount: 350000, ratio: 41.2 }, ... ]
}
```

### 챌린지
```
POST /api/v1/challenges          — 챌린지 생성
GET  /api/v1/challenges/active   — 진행 중 챌린지 조회
GET  /api/v1/challenges/{id}     — 챌린지 상세 (일별 로그 포함)
GET  /api/v1/challenges/history  — 완료/실패 챌린지 목록
```
